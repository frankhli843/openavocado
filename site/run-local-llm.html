Quick Start › Run locally

🖥️ Run with your own local LLM ready

The privacy-first, self-hosted path. Point Open Avocado at your own local model instead of a hosted provider, so learning content never leaves your machine. Any OpenAI-compatible local server works — llama.cpp server, Ollama, vLLM, or LM Studio.

Hosted API key vs. your own local LLM

Open Avocado gives you two standalone ways to bring a model, and they are deliberately separate:

  • Hosted API key — the fastest way to start. You bring an OpenAI or Google AI Studio key and the app calls the provider directly. Prompts leave your machine for the provider. See Run with a direct API key.
  • Your own local LLM (this page) — the privacy / offline / self-hosted path. You run a model on your own hardware and Open Avocado talks to it over its OpenAI-compatible HTTP endpoint. No cloud key, and learning content stays local.

For richer, multi-step agentic authoring there are also the OpenClaw and Hermes agent-runtime paths.

Who this is for

  • You care about privacy or need to work offline — learning data must stay on your own hardware.
  • You already run, or can run, a local model server that speaks the OpenAI chat-completions API.
  • You want to avoid per-token cloud costs and API keys entirely.

Prerequisites

  • A local model server reachable over HTTP that exposes an OpenAI-compatible /v1 endpoint (see the table below).
  • A machine that can serve your chosen model at a usable speed. A small instruct model (roughly 4–8B, quantized) runs on a modern laptop or a single consumer GPU; larger models give better lessons but need more VRAM/RAM.

Configuration

Open Avocado reads the local endpoint from environment variables. Set the base URL (the app appends /chat/completions itself), the model name your server serves, and switch the completion adapter to local-queue so generation is enqueued locally:

# OpenAI-compatible base URL of your local server (NOT including /chat/completions):
export AVOCADOCORE_ACP_ENDPOINT=http://127.0.0.1:8080/v1
# The model name your server exposes (must match your runtime, e.g. an Ollama tag):
export AVOCADOCORE_ACP_MODEL=your-local-model
# Only if your local server requires a bearer token (most do not):
# export AVOCADOCORE_ACP_TOKEN=your-token

# Enqueue lesson generation locally instead of the no-op default:
export AVOCADOCORE_COMPLETION_ADAPTER=local-queue

Replace the host, port, and model with your server's values. Everything runs against 127.0.0.1 for a single-workstation setup; nothing here assumes any particular machine or vendor. See Configuration for the full environment-variable reference.

Run it

# 1. Start your local model server (pick one; substitute your own model/port):
#    llama-server -m model.gguf --port 8080     # llama.cpp
#    ollama serve                               # Ollama (then: ollama pull <model>)
#    vllm serve your-model                      # vLLM
#    lms server start                           # LM Studio
# 2. Point Open Avocado at it with the env vars above, then:
pnpm install
mkdir -p data
pnpm db:migrate --seed
pnpm dev

Smoke test

# 1. Your server answers the OpenAI-style models list:
curl -s http://127.0.0.1:8080/v1/models | head -c 300

# 2. A raw chat completion round-trips (this is the exact API Open Avocado uses):
curl -s http://127.0.0.1:8080/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"your-local-model","messages":[{"role":"user","content":"Reply with the word ok."}]}' \
  | head -c 300

# 3. Open Avocado reports the app healthy:
curl -s http://localhost:3000/api/health | head -c 300

Common failure modes

SymptomCause & fix
Connection refusedServer not running or wrong port. Confirm the /v1/models curl above works before starting the app.
Quiz retries fall back to deterministic textThe endpoint was unreachable, timed out, or returned an unexpected shape. Open Avocado waits at most ~8 s for the local model and then degrades gracefully rather than blocking the learner. A slow model or a cold first request commonly trips this — warm the model, or use a smaller/faster one.
Endpoint errors on response_formatOpen Avocado requests JSON output (response_format: json_object). Most servers support it; if yours does not, use a build/flag that enables JSON/grammar-constrained output (llama.cpp and vLLM both support it) or a model/runtime that honors it.
Lessons look shallow or invalidA small local model may not satisfy the full lesson-generation contract (bespoke interactives, synced visuals, hidden tests). Try a more capable model, or use the local model only for feedback/quiz retries and author lessons via OpenClaw.

What runs locally today: instant learner-facing feedback and adaptive quiz rephrasing/retries call your local OpenAI-compatible endpoint, and lesson generation is enqueued through the local-queue adapter. Full end-to-end lesson authoring from a single local model is best-effort and depends on the model's capability — the same single-model tradeoff as the hosted API-key path. For the richest, multi-step lessons, use a capable model or an agentic runtime.

Privacy & performance tradeoffs

  • Privacy: with a local endpoint, prompts and learning content go only to 127.0.0.1. No cloud provider, no API key, no per-token billing. This is the reason to choose this path over the hosted key.
  • Performance: latency and lesson quality track your hardware and model. Small quantized models are fast but produce thinner lessons; larger models are richer but slower and need more memory. The ~8 s quiz-retry timeout means very slow setups will fall back to deterministic behavior more often.
  • Portability: switching between local and hosted is only a change of base URL and model — no code changes, because every runtime speaks the same OpenAI-compatible contract.

How it connects to lesson generation

Your local server is just an OpenAI-compatible backend for the same feedback and generation code paths described in Configuration and Architecture. Swapping a hosted provider for your own model is a matter of changing the base URL and model — the rest of Open Avocado is unchanged.
