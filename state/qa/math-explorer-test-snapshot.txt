  uid=1_83 heading "What an embedding is" level="3"
  uid=1_84 StaticText "An embedding is a vector: a short list of numbers that represents one token inside the model. In this lesson the vector has 8 visible numbers so you can read it. A real model may use hundreds or thousands of numbers per token."
  uid=1_85 StaticText "Embedding matrix"
  uid=1_86 StaticText "A learned table of model weights. Rows correspond to tokenizer vocabulary IDs. Columns are hidden dimensions, the numeric features the model can adjust during training."
  uid=1_87 heading "How it connects to the tokenizer" level="3"
  uid=1_88 StaticText "The tokenizer does not create meaning. It maps text pieces to stable integer IDs from the vocabulary. The model has an embedding matrix with one row for each possible ID. If the tokenizer outputs ID 14224, the model reads row 14224 from that matrix. That row is the token's embedding."
  uid=1_89 StaticText "A tiny version of the table"
  uid=1_90 StaticText "Imagine a vocabulary with rows 0, 1, 2, 3 and a hidden width of 4. The matrix is just a grid: row 0 has four learned numbers, row 1 has four learned numbers, and so on. Token ID 2 means select row 2. The selected row might look like [0.41, -0.08, 0.73, 0.19]."
  uid=1_91 StaticText "Where embeddings come from: they start as random model weights before training. During training, errors nudge the rows so tokens used in similar contexts develop useful numeric patterns. The tokenizer supplies the address; training shapes the row."
  uid=1_92 heading "Position information keeps order alive" level="3"
  uid=1_93 StaticText "After lookup, the model adds a position vector with the same shape as the embedding vector. Without this, repeated tokens with the same ID would have identical starting hidden states, and the model would not know which occurrence came first."
